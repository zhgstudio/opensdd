# OpenSDD 技能开发 — AI 智能体入口指引

本文档面向**修改 opensdd 技能本身**的 AI 智能体（而非使用该技能的下游项目）。在提出修改或新增功能前，请先阅读以下已决澄清，避免提出已被讨论确认的设计变更。

---

## 已决澄清

以下为经人类评审后确认的设计决策，不再讨论变更：

### AGENTS.md 的定位

AGENTS.md 是**全局入口指引**，承载项目级工程规则与约束（文件操作范围、提交规范、测试要求、升级条件、跨模块规则、任务规范）。存在时各阶段 AI 角色均自动加载——既是 OpenSDD 各阶段的输入（各角色从中了解工程约束），也是 OpenSDD 各阶段的输出（多角色增量写入），最终锁定后作为编码阶段的直接入口。尚未生成时不读取。

### 会话隔离

不为此设计复杂的检查清单或自动化方案。人类知道如何新建对话或重置上下文。

### 阶段三串行

串行是方法论的有意选择，不是性能瓶颈问题。不引入并行设计或归并评审机制。模块按依赖顺序逐个串行推进，一次只设计一个模块。

### 文档语言约束范围

要求统一的是文档的**主体叙述语言**（行文、描述、解释性文字）。标识符（变量名、函数名、类名）、代码片段、URL、专有名词、API 路径等不受此约束，允许与主体叙述语言不同。

### Designer 不写 AGENTS.md

AGENTS.md 承载的是**工程性规则和跨文档关系规范**（文件操作范围、提交规范、测试要求、升级条件、跨模块规则、任务规范），不是细节规则。模块级细节放在对应模块的 `DESIGN.md` 中，公共级细节放在 `ARCHITECTURE.md` 中。

### PLAN.md 不涉及方案细节

PLAN.md 纯粹是任务跟踪表——仅引用 `{NN}-F{NNN}` 编号和标题，标记完成状态。feature 的拆分和方案正确性由架构师和设计师在 `DESIGN.md` 中完成，PLAN.md 不承担描述"怎么做"的职责。

### CI 只测一个主流 Node 版本

CI 不做多版本矩阵测试，固定一个主流 LTS 版本（如 Node 22）即可。

### docs/ 的 .gitignore 排除

当前仓库的 `docs/` 仅包含技能开发计划文档（`docs/superpowers/`），排除是合理的。技能的实际产出文档（SPEC.md、ARCHITECTURE.md、PLAN.md、AGENTS.md 等）在安装该技能的下游项目的 `docs/` 下，不受本仓库 `.gitignore` 影响。

### 模块退役协议

模块不再需要时的处理规则：

1. **物理删除**：如果模块的代码和设计文档均无保留价值，直接删除模块目录，清理所有引用痕迹（ARCHITECTURE.md 模块引用表、模块依赖矩阵、PLAN.md 中的任务、AGENTS.md 中的相关规则）
2. **物理保留**：如果模块代码需保留供参考（如被替换的历史实现），删除 ARCHITECTURE.md 模块引用表中该模块的行、清理 PLAN.md 中的任务引用，并在 AGENTS.md 中说明该模块已退役及其保留原因
3. 不允许使用 `DEPRECATED` 标记——该术语表示"未来将废弃但仍在用"，不适用于已明确退役的场景
4. 模块编号不可复用——退役模块的编号永久保留为其历史标识，新模块只能取后续编号

### AGENTS.md 的初始化与更新

AGENTS.md 可在任意阶段被创建或更新——人类可主动生成初始文件，AI 也可根据需要在 OpenSDD 各阶段以及后续编码阶段中更新其内容。不存在"必须在阶段二才能首次创建"的约束。其核心定位始终是：AI 开发工具了解整个项目的入口，以及项目进行过程中记录工程化规则的状态存储文件。一旦存在，各阶段 AI 角色自动最先加载它。

### 语言一致性不做自动检查

文档语言由人类在阶段一启动时指定。不做自动化脚本级的严格检查——误报成本高于收益。`LANGUAGE_CONSISTENCY` 检查项已从 `opensdd-check` 中移除。

### 阶段二允许 [TBD] 接口标记

如果架构师在阶段二无法确定某模块接口的具体签名，允许在模块依赖矩阵的"所需接口"列中使用 `[TBD]` 标记（如 `[TBD: 角色鉴权接口]`），表示该接口签名待阶段三设计时确认。`[TBD]` 标记不得出现在最终定稿的 ARCHITECTURE.md 中——所有接口必须在阶段四之前确定。详见 `phase-2.md` 中的"接口先行约定"章节。

### 不引入流程退出/中止路径

OpenSDD 方法论是单向推进的线性流程。遇到需求变更或设计返工时，由人类决策：是否回退、从何处继续、修改什么。不为非正常终止设计专门协议。

### 不引入增量模式

`opensdd-check` 维持全量扫描。OpenSDD 阶段不突出追求性能，增量模式带来的复杂度不必要。

### 不做端到端自动化流程测试

OpenSDD 的四阶段是人类参与的审查驱动过程，不适合自动化全流程验证。技能本体的质量保障通过以下自检测试实现（而非集成测试）：
- SKILL.md frontmatter 完整性验证（name、description、metadata.author、metadata.version）
- SKILL.md version 与 package.json version 的一致性检查
- SKILL.md 中引用的 phase 文件路径存在性检查

### public-design-compliance 维持现状

基于正则的命名规范检查（camelCase vs PascalCase）对复杂标识符存在误判率。当前策略"宁可漏检不可误报"是正确权衡，维持现状。如发现更好的方案（精确度接近 100% 且误报率为零），可重新考虑，否则不修改。

### 仓库结构说明不在 AGENTS.md

本仓库的目录拓扑和文件职责划分说明（如根级 AGENTS.md 与下游项目 AGENTS.md 的区别）放在 `README.md` 中。AGENTS.md 聚焦于工程约束与已决澄清，不承担仓库结构概述职责。
