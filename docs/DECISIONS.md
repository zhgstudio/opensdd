## 已采纳

以下为经评审确认的设计决策，不逆（不再讨论变更）。开始审查前请先阅读，避免提出已被确认的设计变更。

### AGENTS.md 的定位

AGENTS.md 是**全局入口指引**，承载项目级工程规则与约束（文件操作范围、提交规范、测试要求、升级条件、跨模块规则、任务规范）。存在时各阶段 AI 角色均自动加载——既是 OpenSDD 各阶段的输入（各角色从中了解工程约束），也是 OpenSDD 各阶段的输出（多角色增量写入），最终锁定后作为编码阶段的直接入口。尚未生成时不读取。

### 会话隔离

不为此设计复杂的检查清单或自动化方案。人类知道如何新建对话或重置上下文。

### 文档语言约束范围

要求统一的是文档的**主体叙述语言**（行文、描述、解释性文字）。标识符（变量名、函数名、类名）、代码片段、URL、专有名词、API 路径等不受此约束，允许与主体叙述语言不同。

### Designer 不写 AGENTS.md

AGENTS.md 承载的是**工程性规则和跨文档关系规范**（文件操作范围、提交规范、测试要求、升级条件、跨模块规则、任务规范），不是细节规则。模块级细节放在对应模块的 `DESIGN.md` 中，公共级细节放在 `ARCHITECTURE.md` 中。

### PLAN.md 不涉及方案细节

PLAN.md 纯粹是任务跟踪表——仅引用 `{MODULE}-F{NNN}` 编号和标题，标记完成状态。feature 的拆分和方案正确性由架构师和设计师在 `DESIGN.md` 中完成，PLAN.md 不承担描述"怎么做"的职责。

### CI 只测一个主流 Node 版本

CI 不做多版本矩阵测试，固定一个主流 LTS 版本（如 Node 22）即可。

### 模块退役协议

模块不再需要时的处理规则：

1. **物理删除**：如果模块的代码和设计文档均无保留价值，直接删除模块目录，清理所有引用痕迹（ARCHITECTURE.md 模块引用表、模块依赖矩阵、PLAN.md 中的任务、AGENTS.md 中的相关规则）
2. **物理保留**：如果模块代码需保留供参考（如被替换的历史实现），删除 ARCHITECTURE.md 模块引用表中该模块的行、清理 PLAN.md 中的任务引用，并在 AGENTS.md 中说明该模块已退役及其保留原因
3. 不允许使用 `DEPRECATED` 标记——该术语表示"未来将废弃但仍在用"，不适用于已明确退役的场景
4. 模块编号不可复用——退役模块的编号永久保留为其历史标识，新模块只能取后续编号

### AGENTS.md 的初始化与更新

AGENTS.md 可在任意阶段被创建或更新——人类可主动生成初始文件，AI 也可根据需要在 OpenSDD 各阶段以及后续编码阶段中更新其内容。不存在"必须在阶段二才能首次创建"的约束。其核心定位始终是：AI 开发工具了解整个项目的入口，以及项目进行过程中记录工程化规则的状态存储文件。一旦存在，各阶段 AI 角色自动最先加载它。

### 语言一致性不做自动检查

文档语言由人类在阶段一启动时指定。不做自动化脚本级的严格检查——误报成本高于收益。`LANGUAGE_CONSISTENCY` 检查项已从 `opensdd-check` 中移除。

### 阶段二允许 [TBD] 接口标记

如果架构师在阶段二无法确定某模块接口的具体签名，允许在模块依赖矩阵的"所需接口"列中使用 `[TBD]` 标记（如 `[TBD: 角色鉴权接口]`），表示该接口签名待阶段三设计时确认。`[TBD]` 标记不得出现在最终定稿的 ARCHITECTURE.md 中——所有接口必须在阶段四之前确定。详见 `phase-2.md` 中的"接口先行约定"章节。

### 不引入流程退出/中止路径

OpenSDD 方法论是单向推进的线性流程。遇到需求变更或设计返工时，由人类决策：是否回退、从何处继续、修改什么。不为非正常终止设计专门协议。

**大规模回退指引**：当阶段三进行中发现阶段二的架构决策存在结构性缺陷，影响范围超过 3 个模块或涉及核心数据流变更时，设计反哺协议的单点暂停机制可能不足以应对。此时建议人类：

1. 回退至阶段二，修复 `ARCHITECTURE.md` 中的架构决策
2. 将受影响的模块标记为"待重新设计"，而非逐模块修补
3. 重新定稿阶段二后，按变更传播协议级联更新阶段三的受影响模块

### 不引入增量模式

`opensdd-check` 维持全量扫描。OpenSDD 阶段不突出追求性能，增量模式带来的复杂度不必要。

### 不做端到端自动化流程测试

OpenSDD 的四阶段是人类参与的审查驱动过程，不适合自动化全流程验证。技能本体的质量保障通过以下自检测试实现（而非集成测试）：
- SKILL.md frontmatter 完整性验证（name、description、metadata.author、metadata.version）
- SKILL.md version 与 package.json version 的一致性检查
- SKILL.md 中引用的 phase 文件路径存在性检查

### public-design-compliance 已移除

`public-design-compliance` 基于关键词启发式的语义检查存在固有限度——硬编码的章节标题匹配和标识符命名规则无法覆盖实际项目中的多样措辞，且误报成本高于收益。该检查已于 2026-06-20 移除。统一的命名规范由人类在审查时保证，不做自动化脚本级检查。

### 仓库结构说明不在 AGENTS.md

本仓库的目录拓扑和文件职责划分说明（如根级 AGENTS.md 与下游项目 AGENTS.md 的区别）放在 `README.md` 中。AGENTS.md 聚焦于工程约束与已决澄清，不承担仓库结构概述职责。

### 不引入残留文件自动检查

不引入 `NO_GARBAGE` 这类自动化残留文件检测。残留文件（`_v2.md`、`_final.md`、`.bak` 等）由人类在阶段规范中约束 AI 避免生成即可，不需要自动化脚本级的检测。自动化检查在此场景下收益低、复杂度高。原 `garbage.js` 已于 2026-06-20 移除。

### 下游项目 AGENTS.md 的位置

下游项目的 AGENTS.md 位于项目**根目录**（而非 `docs/` 下）。SKILL.md 和 README 中的核心目录拓扑中 AGENTS.md 展示在根级，与 AGENTS.md 为 AI 开发工具自动加载的入口文档定位一致。

### DECISIONS.md 在核心拓扑中标注为可选

DECISIONS.md 是可选文档，不视为 OpenSDD 核心必产物。已在 SKILL.md 和 README 的核心目录拓扑中标注"（可选）"，不排在 5 类核心文档之列。

### 变更传播协议仅适用于下游项目

变更传播协议（SKILL.md 中定义）针对的是下游项目对已定稿产物的修改场景（SPEC → ARCH → API/DESIGN → PLAN 的级联更新）。本技能开发项目自身的方法论或工具变更不走该协议——直接修改文件后执行正常 CI test 流程即可。

### CI 不运行 opensdd-check 对自身

opensdd-check 是面向下游项目的结构合规性校验工具。本仓库 CI 只执行 test（含 lint、format-check、单元测试）来保障技能开发质量，不调用 opensdd-check 自举。

### 临时过程文档不保留在版本管理中

过时的实施计划、设计草案等临时过程文档在定稿后即删除，不留存在仓库中（包括 `docs/` 下）。相关决策已在最终产物（SKILL.md、phase 文件、工具代码）中体现，不需要保留历史草案作为参考。

### 版本一致性自动检查

`VERSION_CONSISTENCY` 已作为 opensdd-check 的内置检查项。验证三项版本号完全一致：SKILL.md `metadata.version`、根 `package.json` version、`opensdd/opensdd-check/package.json` version。不一致时 fail；package.json 缺失时 warn；SKILL.md 不存在时 skip。支持点记法（`metadata.version: X.Y.Z`）和嵌套 YAML（`metadata:\n  version: X.Y.Z`）两种 frontmatter 格式。

### CRLF 兼容性统一处理

所有硬编码的 `content.split('\n')` 已全部替换为 `lib/line-split.js` 中导出的共享 `splitLines(content)` 函数（内部使用 `content.split(/\r?\n/)`），覆盖 8 个检查文件（matrix.js、agents.js、plan.js、tbd-residual.js、strategies/{http,grpc,function}.js、version-consistency.js）。后续新增文件也应使用该工具函数而非直接调用 `split('\n')`。

### tmp/ 目录清理检查

`NO_TMP` 已作为 opensdd-check 的内置检查项。检测 `docs/` 目录下是否存在 `tmp/` 目录（临时过程文档未清理的迹象）。存在时 fail。已在 `finalization.md` 的自动化验证表中取代已移除的 `NO_GARBAGE`。

### 公共设计栅栏规则

在 `phase-2.md` 中新增"公共设计栅栏规则"章节，列出 6 类必须进入 `ARCHITECTURE.md` 公共设计、不得下沉到模块 `DESIGN.md` 的决策类型：全局错误格式、分页规范、时区与时间格式、审计字段、跨模块通信协议、日志与监控。架构师以此清单为下限而非上限。清单外的模块特有细节留在对应 `DESIGN.md` 中。

### 替代方案记录格式简化

`phase-3.md` §8 中"一段简要备忘记录方案名称、核心思路和未被采纳的核心理由"改为"仅记录方案名和核心拒绝理由"——减少 AI 在"简要"和"过于简略"之间摇摆的空间。对应的方法学审视检查清单项已更新。

### NO_TMP 跳过 node_modules/.git

`NO_TMP` 检查（`checks/no-tmp.js`）的 `findTmpDirs` 递归搜索中跳过 `node_modules` 和 `.git` 目录，防止第三方包中的 `tmp/` 目录导致误报。排除目录按需扩展。

### PLAN.md depends 跨行存在性验证

`PLAN_FORMAT` 检查（`checks/plan.js`）新增依赖 ID 存在性验证——`depends: T-NNN` 引用的任务 ID 必须在 PLAN.md 中有对应定义行，否则 fail。此前仅验证语法格式。

### 零价值测试文件清理

`test-index.js`（仅验证模块加载为 `function` 类型——JavaScript require 级的最弱断言）已合并入 `test-smoke.js` 后删除。后续新增测试应直接写入有意义的边界或集成测试中。

### config.js 数组合并改为用户替换默认

此前 `mergeConfig` 将用户配置的数组字段追加到默认值之后（`[...defaults, ...user]`），导致用户无法通过显式设置缩小 `requiredFiles`、`requiredApiSections` 等数组的范围。改为用户数组完全替换默认数组。

### decisions.js 移除"理由"/"取消条件"区段语义检查

DECISIONS.md 的"理由"和"取消条件"是语义性的内容要求，不适合用脚本进行确定性硬编码匹配。已移除该区段检查，保留 frontmatter 格式验证。

### 编号体系显式标注三位数能力上限

在 SKILL.md 编号体系章节补充了三位数字编号的范围约束：REQ-NNN（000-999）、{MODULE}-FNNN（000-999）、T-NNN（000-999）。此范围为能力上限，不支持更大编号，超大项目应拆分为多个子项目。

### PM Agent 读取 AGENTS.md（阶段一）

`phase-1.md` 上下文说明已更新：如果项目已存在 `AGENTS.md`，PM Agent 应读取以了解项目背景和已有约束。因主流 AI 开发工具（如 OpenCode）会自动加载 `AGENTS.md` 作为入口指引，阶段一不排除此文件。尚未生成时不阻塞流程。

### ESLint 8.x → 9.x 迁移

已完成迁移：`.eslintrc.json`（旧格式）替换为 `eslint.config.js`（flat config），`eslint-plugin-jsdoc` 同步升级至兼容版本。详见 opensdd/opensdd-check/ 的 eslint.config.js。

### Node.js 引擎最低版本更新至 22

Node 18 已于 2025-04 终止生命周期（End of Life），根 `package.json` 的 `engines.node` 从 `>=18` 更新至 `>=22`。CI 已使用 Node 22。
`opensdd/opensdd-check/package.json` 已于 2026-06-21 对齐为 `>=22`。

### REQ→Feature 追溯的 warn-only 定位确认

TRACEABILITY 检查（`checks/traceability.js`）对需求到特性的追溯仅为 warn 级别，不阻断 CI。此定位是方法论设计的合理缺口——并非所有需求都可映射为独立特性，并非所有特性都对应编号需求。自动化做结构性硬约束的误报率不可接受，应依赖人类在阶段四和最终定稿时评审判断。

### 阶段三横切关注点上浮机制

设计师在阶段三发现新的跨模块共性模式（未被 `ARCHITECTURE.md` 公共设计或公共设计栅栏规则覆盖时），应通过设计反哺协议暂停→反馈→由人类裁定是否纳入公共设计。已在 `phase-3.md` 的设计反哺协议章节中明确文档化。

### 阶段四 sub-agent 任务规划隔离

多模块项目中，PM Agent 单会话加载全部模块设计进行任务拆分的认知负载过高。建议 PM Agent 主会话负责跨模块一致性校验和最终整合，每个模块的任务拆分由独立的 sub-agent 会话完成。已在 `phase-4.md` 中新增"会话管理"章节。

## 已拒绝

### 阶段三引入并行或批量评审

- **提出场景**：2026-06-20 代码评审中建议为无依赖关系的模块引入批量提交评审模式，以缓解串行吞吐量瓶颈。
- **理由**：每个模块评审结果需要人类审查，一次性输出多个评审结果会让人类忙不过来。OpenSDD 阶段不追求速度，更注重过程做扎实，保证产物质量。该原则已在已决原则中记录。
- **取消条件**：无。此为已决设计原则，不因规模变化而回退。

### 会话隔离自动化工具

- **提出场景**：2026-06-20 代码评审中指出当前会话管理完全依赖人类手动操作，建议开发自动化工具辅助。
- **理由**：人类的基本经验能保证会话隔离的有效性。该事项已在已决原则中记录，不引入复杂的自动化方案。
- **取消条件**：无。

### opensdd-check 输出国际化

- **提出场景**：2026-06-20 代码评审中指出 opensdd-check（warnings、errors、help）为英文输出，与方法论支持中文文档语言存在不一致。
- **理由**：opensdd-check 输出只考虑英文，不考虑其他语言。
- **取消条件**：无。

### 使用 TypeScript 重构 opensdd-check

- **提出场景**：2026-06-20 代码评审中建议将 opensdd-check 从 JavaScript 迁移到 TypeScript，以增强契约校验和开发时类型检查。
- **理由**：重构工作量过大，且会引入额外基础设施（tsconfig、编译步骤、CI 调整）。当前检查工具体量不大（约 30 个模块文件），JavaScript + JSDoc 已能提供足够的类型文档。
- **取消条件**：当 opensdd-check 规模显著增长（如超过 50 个模块文件）或需要作为独立 npm 包发布时重新评估。

### 接口草图机制

- **提出场景**：方法学审视中建议在阶段二引入轻量级接口草图（接口签名+数据流），以提前暴露依赖问题。
- **理由**：`[TBD]` 标记已覆盖"知道不确定"的场景，设计反哺协议覆盖"以为确定但错了"的场景。引入草图模糊了阶段二（架构）与阶段三（详细设计）的职责边界，增加阶段二的产出要求，与"架构师不做模块内部设计"的原则冲突。
- **取消条件**：如果在实际使用中发现 `[TBD]` 标记 + 设计反哺协议无法覆盖大部分接口争议场景（如 >30% 的模块在阶段三出现接口变更），可重新评估。

### 量化设计反馈阈值

- **提出场景**：建议为设计反哺协议设置定量触发阈值（如"连续 3 个模块出现反哺则暂停流程"）。
- **理由**：30% 阈值在不同规模的项目（5 模块 vs 50 模块）中含义完全不同，统一阈值缺乏可操作意义；反馈频率本身就是自然信号，人类看到"又出问题了"会自然决定是否暂停，不需要预设阈值。
- **取消条件**：如果某个大型项目在阶段三中频繁触发反哺导致人类难以跟踪，可引入项目级反哺计数器作为人类辅助信息，但不作为自动暂停条件。

### 阶段二包含 sketch 级接口定义

- **提出场景**：建议阶段二架构师不仅声明模块依赖，还给出各模块接口的 sketch 级定义（大致参数和返回结构）。
- **理由**：与"接口草图机制"本质上是对同一问题的重复提议（详见上文），只是侧重点从依赖验证转向模块接口预定义。同样会模糊阶段二/三边界，且可能导致阶段二产生大量在阶段三被推翻的伪接口定义。
- **取消条件**：取消条件同上。

### 编码阶段引入 DECISIONS.md

- **提出场景**：建议在编码阶段也引入决策记录，记录 AI 在编码过程中的关键技术决策。
- **理由**：编码阶段的决策记录属于下游项目的开发规范，不在 OpenSDD 技能本体的约束范围内。OpenSDD 只覆盖编码前规范阶段，不对编码阶段的过程管理做规定。编码阶段是否引入 DECISIONS.md 应由下游项目自行决定。
- **取消条件**：如果 OpenSDD 未来扩展覆盖到编码阶段管理，则重新评估。

### 文档自版本号 + 跨文档依赖版本一致性检查

- **提出场景**：建议每个文档开头添加自身版本号（x.y.z）及所依赖文档的名称与版本号，每次修订需更新版本号，opensdd-check 检查依赖版本一致性，不一致时 warning。
- **理由**：变更传播协议已覆盖该场景，规则不缺缺的是执行纪律；文档变更频率高于代码，版本号疲劳将导致开发者无视检查；大量文档修订不要求下游变更（如 SPEC 修正拼写），误报率预计很高，人力成本从变更时间转移到判断时间，净收益为零或负；依赖图涉及传递级联，实现复杂度与收益不成正比。
- **取消条件**：如果在实际使用中发现变更传播协议因缺少自动化兜底导致频繁的漏传播事故（如 >20% 的架构变更未同步到下游 DESIGN.md），可重新评估。

### 阶段三设计师不拆分 SPEC.md 加载

- **提出场景**：2026-06-21 方法学审视中建议设计师按模块过滤 SPEC.md 内容，只加载与当前模块相关的需求部分，避免设计师每次会话解析不相关需求文本。
- **理由**：需求文档体量不会大到需要拆分，且引入按模块过滤会增加项目文档拓扑复杂性。设计师阅读完整的 SPEC.md 是可接受的认知负担。
- **取消条件**：无。

### 阶段二不记录架构替代方案

- **提出场景**：2026-06-21 方法学审视中建议在 `phase-2.md` 增加可选的架构替代方案记录章节，为技术栈、模块划分等决策保留推导背景。
- **理由**：架构决策稳定性强于模块设计，且 `ARCHITECTURE.md` 全量加载到每个模块设计的上下文中，记录被拒绝的方案会干扰上下文注意力。如有必要，可记录到 `DECISIONS.md` 中。
- **取消条件**：无。

### 设计质量自动化检查

- **提出场景**：2026-06-30 方法学审视中建议为设计质量引入自动化检查机制。
- **理由**：设计质量由 AI + 人类审查判定，不适合也不应该由自动化工具判断。opensdd-check 的结构性检查（文件存在性、接口签名一致性、编号格式）已覆盖可自动化的部分，设计方案本身的好坏是人类的判断范畴。
- **取消条件**：无。

### 模块粒度量化定义

- **提出场景**：2026-06-30 方法学审视中建议在方法论中明确定义模块粒度的量化标准（如单模块不应超过 X 个特性）。
- **理由**：模块粒度由 AI 根据专业经验和项目上下文确定，不适合也不可能由固定的方法论规则量化。不同项目、不同领域的最佳粒度天然不同，强行量化会产生错误的刚性约束。
- **取消条件**：无。

## 遗留暂不处理

### opensdd-check 性能基准

- **提出场景**：2026-06-20 代码评审中指出 interface-consistency 检查在大型项目（20+ 模块）中可能成为瓶颈，建议建立性能基准。
- **理由**：当前所有检查基于文件系统，在项目模块数量有限时性能足够。引入基准测试的收益在当前阶段不明确。
- **取消条件**：当实际使用中遇到可感知的性能瓶颈（如单次运行超过 5 秒）时重新评估。

### frontmatter 解析逻辑重复

- **提出场景**：2026-06-20 方法学审视中指出 `version-consistency.js` 与 `frontmatter.js` 均实现了 YAML frontmatter 正则切分逻辑（`---` 边界匹配 + 字段路径解析），存在重复。
- **理由**：当前工具规模约 30 个模块文件，重复的 ~10 行代码维护成本低。抽象到 `lib/` 的复杂度收益比在工具规模显著增长前不划算。
- **取消条件**：当 opensdd-check 模块文件数超过 50 个或 frontmatter 解析逻辑发生第三次复用节点时重新评估。
