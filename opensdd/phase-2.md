# 阶段二：总体架构设计（Architect Agent）

## 角色

架构师（Architect Agent）

## 上下文

`docs/SPEC.md`（只读）。`AGENTS.md` 加载规则见 SKILL.md「角色职责边界」章节。

## 输入

定稿的 `docs/SPEC.md`

## 输出

- `docs/ARCHITECTURE.md`
- 写入 `AGENTS.md` 主体

## 行为

1. 以 `SPEC.md` 为唯一上下文，使用人类指定语言生成 `docs/ARCHITECTURE.md`
2. `ARCHITECTURE.md` 必须包含以下章节：

   ### 技术栈标准
   明确全系统技术栈、框架版本、构建工具、包管理方式。

   ### 全局编码规范
   命名风格（如 camelCase）、文件组织结构、错误返回格式（Error Envelope）、日志规范。

   ### 模块引用表
   以表格列出所有模块及其两位数字编号，格式：
   ```markdown
    ## 模块引用表
    | 编号 | 模块名 | 功能简述 | 详细设计 |
    |------|--------|----------|----------|
     | 01 | auth | 用户认证 | docs/modules/01-auth/ |
     | 02 | task-core | 任务核心 | docs/modules/02-task-core/ |
     ```
    - **编号** = 两位数字，与模块目录名一致
    - **模块名** = 英文短名，与目录名一致
    - **详细设计** = 指向该模块目录的路径（目录下包含 API.md 和 DESIGN.md 两个文件）

   ### 模块依赖矩阵
   ```markdown
   ## 模块依赖矩阵
   | 模块 | 依赖 | 所需接口 |
   |------|------|----------|
   | 02-task-core | 01-auth | POST /auth/verify |
   | 03-api-gateway | 01-auth, 02-task-core | … |
   ```

    ### 公共设计
    跨模块共用的设计决策：认证方案、数据流拓扑、部署架构概览（如适用）。

    ### 公共设计栅栏规则

    以下决策类型必须进入 `ARCHITECTURE.md` 的"公共设计"章节，不得下沉到模块 `DESIGN.md` 中重复约定：

    - **全局错误格式**：Error Envelope 结构（如 `{ code, message, details? }`）、HTTP 状态码映射策略
    - **分页规范**：分页参数名（`page`/`size`/`cursor`）、响应格式、默认值与上限
    - **时区与时间格式**：全部使用 UTC / ISO8601 / Unix Timestamp
    - **审计字段**：所有数据实体公共字段命名（如 `created_at`、`updated_by`、`deleted_at`）
    - **跨模块通信协议**：同步/异步选择、限流策略、重试与超时策略
    - **日志与监控**：日志级别划分、Trace ID 传播方式、指标采集点

    不属于以上类别的设计决策不需要提前约定。模块特有的数据结构、业务逻辑、内部流程等细节留在对应的 `DESIGN.md` 中。架构师应以此清单为下限而非上限——特定项目如有额外跨模块约定，同样应纳入公共设计。

    ### 接口先行约定

    模块依赖矩阵中的"所需接口"列是架构师在阶段二对模块间契约的**预期声明**。如果某个模块的接口在阶段二尚未完全确定（依赖方或提供方的具体签名待阶段三设计时定稿），可以使用 `[TBD]` 标记：

    ```markdown
    ## 模块依赖矩阵
    | 模块 | 依赖 | 所需接口 |
    |------|------|----------|
    | 02-task-core | 01-auth | POST /auth/verify, [TBD: 角色鉴权接口] |
    ```

    `[TBD]` 标记的含义与处理规则：
    - 标志着该接口的具体签名**待阶段三设计时确认**，不是最终承诺
    - 阶段三设计师在进行到受影响的模块时，必须优先确认 `[TBD]` 接口的签名
    - 确认后的签名通过**设计反哺协议**反馈给阶段二（如需更新 ARCHITECTURE.md）
    - `[TBD]` 标记不得出现在最终定稿的 ARCHITECTURE.md 中——所有接口必须在阶段四之前确定

3. **重要职责边界**：`ARCHITECTURE.md` 只写整体架构和公共设计，具体的模块内部设计放在对应模块的 `DESIGN.md` 中。`ARCHITECTURE.md` 中通过模块引用表引用到各模块目录，这样编码阶段的开发者只需要阅读 `AGENTS.md` + `ARCHITECTURE.md`（公共部分）+ 当前模块目录下的 `API.md` 和 `DESIGN.md`。`AGENTS.md` 载入的是项目级工程规则与约束（文件操作范围、提交规范、测试要求、升级条件、跨模块规则等），OpenSDD 各阶段 AI 角色均须读取（存在时）。

4. **写入 `AGENTS.md` 主体**，以 `## ` 为章节标题追加以下章节（AGENTS.md 的顶层章节须使用 `## ` 二级标题，子层级允许使用 `### ` 三级标题，以确保 `opensdd-check` 的 AGENTS_SECTIONS 检查项能正确识别顶层章节）：
   - 文件操作范围（AI 可读/可写的目录白名单）
   - 提交规范（Commit 消息格式）
   - 测试要求（最小覆盖率、必须通过的测试范围）
   - 升级条件（什么情况下暂停并请求人类介入）
   - 跨模块规则（允许读取的模块接口文件列表等）
    - 模块目录说明：说明 `docs/modules/{NN}-{name}/` 目录下各文件的分工与用途——`API.md` 为模块的对外接口定义（跨模块契约，只读），`DESIGN.md` 为模块的内部实现细节（含 `{MODULE}-F{NNN}` 特性列表），编码阶段开发者需同时阅读两者
   - 决策记录说明（如 `docs/DECISIONS.md` 文件存在）：引用该文件，说明其仅审查/审视/评审过程必须加载，其他阶段不需要加载

## 审查清单

开始评审前，如存在 `docs/DECISIONS.md` 且与当前阶段相关，请先阅读以了解已知遗留决策和已拒绝事项，避免重复提出。

人类评审 ARCHITECTURE.md 时建议逐项核对以下清单：

- [ ] **技术栈标准**：是否完整明确了语言、框架、版本、构建工具、包管理器
- [ ] **模块引用表**：所有模块是否都有唯一两位编号 + 英文短名，指向路径是否存在
- [ ] **模块依赖矩阵**：依赖关系是否准确，所需接口列是否写明了具体接口签名
- [ ] **全局编码规范**：命名风格、文件组织结构、错误返回格式、日志规范是否明确
- [ ] **公共设计**：认证方案、数据流拓扑、部署概览等跨模块决策是否有记录
- [ ] **AGENTS.md 主体**：文件操作范围、提交规范、测试要求、升级条件、跨模块规则是否均已写清
- [ ] **模块目录说明**：API.md 与 DESIGN.md 的分工是否已在 AGENTS.md 中解释清楚
- [ ] **决策记录引用**：如项目存在 `DECISIONS.md`，AGENTS.md 中是否已引用并说明其加载规则
- [ ] **编号规则**：模块编号是否递增追加，无中间插入或重排

## 晋级条件

人类明确回复"ARCHITECTURE.md 定稿"，并提交 Git
