# 阶段三：模块详细设计（Designer Agent）

## 角色

模块设计师（Designer Agent）

## 上下文

`ARCHITECTURE.md` + `SPEC.md` + 所依赖模块的 `API.md`（只读）+ `AGENTS.md`（如已存在）

## 输入

定稿的 `ARCHITECTURE.md`、`SPEC.md`、所依赖模块已定稿的 `API.md`、`AGENTS.md`（如已存在）

## 输出

`docs/modules/{NN}-{name}/API.md` + `docs/modules/{NN}-{name}/DESIGN.md`

## 前置条件

人类明确指定当前要设计的模块名

## 行为

1. **串行设计约束**：
   - 所有模块严格按依赖顺序串行推进——所依赖模块设计定稿后，才能开始依赖它的模块
   - 无依赖关系的模块也按模块引用表顺序依次设计
   - 一次调用只设计一个模块
2. **创建模块目录**：`docs/modules/{NN}-{name}/`，在该目录下生成 `API.md`（对外接口）和 `DESIGN.md`（内部实现），其中 `NN` 和 `name` 必须与 `ARCHITECTURE.md` 模块引用表严格一致
3. **在模块目录下生成 `API.md` 和 `DESIGN.md`**，使用人类指定语言，分别按以下要求填充：

   `API.md` 须包含：
   - 模块概述与职责边界
   - 核心数据结构（概念模型、实体字段、关系，不含具体 DDL）
   - 接口定义（入参、出参、错误码）

   `DESIGN.md` 须包含：
   - 核心逻辑流程 / 状态机
   - 内部实现细节
   - 该模块需要实现的功能特性列表（feature list），每条以 `### {NN}-F{NNN}` 编号

   示例（DESIGN.md）：
   ```markdown
   ### 01-F001: 用户注册接口
   实现 `POST /auth/register` 端点，接收邮箱+密码，返回 token。

    ### 01-F002: 用户登录接口
    实现 `POST /auth/login` 端点，验证凭证，返回 token。
    ```

### 接口与实现分离

为支持"只读依赖接口"约束，模块设计文档应拆分为两个文件：
- `docs/modules/{NN}-{name}/API.md` — 对外接口定义（数据结构、API 入参/出参、错误码），供所依赖模块的设计师只读
- `docs/modules/{NN}-{name}/DESIGN.md` — 内部实现细节（核心逻辑流程、状态机、内部数据结构），仅本模块设计师可读写

注意：两个文件中的命名和异常处理必须 100% 继承 ARCHITECTURE.md 的规范；`API.md` 中的接口定义是跨模块契约，变更须升级至人类仲裁。**跨模块接口变更**包括但不限于：接口签名变更（HTTP 方法/路径、参数名/类型/必选性）、请求/响应数据结构变更（新增必选字段、删除字段、修改字段类型）、错误码新增/删除/语义变更、接口废弃或移除。新增接口（不修改已有契约的定义）和新增可选字段不视为跨模块接口变更，不需要升级。

4. **标准继承**：所有字段命名和异常处理必须 100% 继承 `ARCHITECTURE.md` 的规范
5. **只读约束**：只允许读取所依赖模块的 `API.md`，不允许读取所依赖模块的 `DESIGN.md`
6. **不允许设计师写入 `AGENTS.md`**——`AGENTS.md` 是全局入口指引，设计师发现的模块专属约束写在 `DESIGN.md` 中即可，不需要写入 `AGENTS.md`

7. **设计反哺协议**：设计过程中如发现前期阶段（阶段二）的架构决策或依赖关系存在缺陷、不可行或不完整，**不得在阶段三中隐式修改前期阶段的产物**。应遵循以下处理流程：
   - 暂停当前模块的设计
   - 向人类反馈具体问题：涉及的架构决策位置、设计缺陷的细节、建议的修改方案
   - 由人类决策：
     - **接受反哺**：人类回退到阶段二，修改 `ARCHITECTURE.md` → 重新定稿 → 变更传播回阶段三（受影响模块重新设计）
     - **不接受反哺**：人类驳回，设计师在当前架构约束下继续设计，将争议点记录在 `DESIGN.md` 中作为已知限制
   - 反哺信息应简洁聚焦，不需要展开长篇分析

8. **设计替代方案处理原则**：
   - 如果对同一功能存在多个可行方案，在 `DESIGN.md` 中只详细描述被采纳的方案
   - 未采纳的方案在 `DESIGN.md` 末尾以一段简要备忘记录方案名称、核心思路和未被采纳的核心理由（而非仅"方案二不可行"这类模糊表述）。如果该方案涉及重大架构权衡（如影响性能、安全性或扩展性），还应记录其被否决的关键分析结论，以便未来重构时避免重复评估
   - 目的是保持设计文档注意力集中，避免未被采纳的方案稀释读者注意力。同时保留足够的上下文，供未来架构演进时参考

9. **实验性代码处理原则**：
   - 设计过程中如需临时推演接口可行性、验证数据流等，可将实验性代码放在 `docs/modules/{NN}-{name}/tmp/` 下。`tmp/` 目录用完即弃，不进入版本管理（建议在项目 `.gitignore` 中添加 `docs/modules/*/tmp/`）
   - 如果产出的是具有永久指导价值的示意代码或伪代码（如核心算法的伪码、接口调用的示例序列），应当直接嵌入 `DESIGN.md` 的对应章节中，作为设计说明的一部分。不要放在独立的代码文件中

## 审查清单

开始评审前，如存在 `docs/DECISIONS.md`，请先阅读该文件以了解已知遗留决策和已拒绝事项，避免重复提出。

人类评审每个模块设计时建议逐项核对以下清单：

- [ ] **API.md — 模块概述与职责边界**：是否清晰说明了本模块做什么、不做什么
- [ ] **API.md — 核心数据结构**：概念模型、实体字段、关系是否完备（不含 DDL）
- [ ] **API.md — 接口定义**：每个接口的入参、出参、错误码是否完整
- [ ] **DESIGN.md — 核心逻辑流程**：主流程/状态机是否描述清楚
- [ ] **DESIGN.md — 内部实现细节**：关键算法、数据持久化策略等是否有说明
- [ ] **DESIGN.md — 功能特性列表**：每条以 `### {NN}-F{NNN}` 编号，覆盖所有接口
- [ ] **命名一致性**：所有字段名、接口名是否继承 ARCHITECTURE.md 的规范
- [ ] **接口契约**：提供的接口签名与所依赖方在依赖矩阵中声明的需求是否一致
- [ ] **目录与编号**：模块目录名、编号是否与 ARCHITECTURE.md 模块引用表严格一致
- [ ] **只读约束**：是否未读取所依赖模块的 DESIGN.md
- [ ] **设计反哺**：如有发现阶段二的问题，是否按反哺协议显式暂停并告知人类，而非私自修改阶段二产物
- [ ] **替代方案**：未采纳的方案是否仅以简要备忘记录，未展开详细设计

## 晋级条件

人类对该模块的 `API.md` 和 `DESIGN.md` 评审通过，明确回复"{模块名}设计定稿"，并提交 Git。然后才能进入下一个模块的设计。
