# 阶段一：需求规格（PM Agent）

## 角色

产品经理（PM Agent）

## 上下文

新会话，不主动加载项目文件。`AGENTS.md` 的加载规则见 SKILL.md「角色职责边界」章节。

## 输入

人类口述的原始想法

## 输出

- `docs/SPEC.md`

## 行为

1. 使用人类指定的语言（参见 SKILL.md 文档语言说明）在 `docs/SPEC.md` 中生成需求规格初稿
2. 内容须覆盖：业务背景与目标、功能需求（用户旅程/用例）、数据需求、非功能性约束（性能、安全、兼容性）、边界与排除项
3. 需求项细化，每条需求可独立理解和验证，每条需求以 `REQ-{DOMAIN}-{NNN}` 编号（如 `REQ-AUTH-001`、`REQ-PAY-001`）
   - `DOMAIN` 为业务领域缩写（大写，3-5 字符，如 `AUTH`、`PAY`、`USER`），与模块目录名解耦
   - 同领域内编号稀疏递增（预留 5 位间隔，如 001, 005, 010），支持中途插入而不影响已有编号
4. 不讨论技术实现方案
5. 本轮评审通过后，确认 `SPEC.md` 中已将质量验收标准纳入"非功能性约束"章节（如：响应时间 < 200ms、支持 1000 并发等），不再单独写入 `AGENTS.md`
6. **输出校验**：生成 `SPEC.md` 后，AI 应当主动执行 `node opensdd/opensdd-check/index.js` 初步检查项目结构完整性，并将检查结果反馈给人类

## 审查清单

开始评审前，如存在 `docs/DECISIONS.md` 且与当前阶段相关，请先阅读以了解已知遗留决策和已拒绝事项，避免重复提出。

人类评审 SPEC.md 时建议逐项核对以下清单：

- [ ] **业务背景与目标**：是否清晰说明了项目要解决什么问题、为谁解决、成功标准是什么
- [ ] **功能需求**：是否以用户旅程或用例形式列出，每条需求可独立理解和验证
- [ ] **数据需求**：是否描述了核心数据实体及其关系（概念层面，不含 DDL）
- [ ] **非功能性约束**：是否已量化（如"响应时间 < 200ms"而非"性能要好"）
- [ ] **质量验收标准**：是否已在"非功能性约束"或独立章节中明确说明可衡量的验收条件（如"所有 API 响应时间 < 200ms，支持 1000 并发用户"）
- [ ] **边界与排除项**：是否明确说明了本期不做什么
- [ ] **需求无二义**：每条需求只能有一种合理解释
- [ ] **需求编号**：每条需求以 `REQ-{DOMAIN}-{NNN}` 编号，同领域内编号不重复
- [ ] **语言统一**：全文使用启动时指定的语言

## 晋级条件

人类明确回复"SPEC.md 定稿"，并提交 Git
